这篇文章分享了 Anthropic Labs 团队在两个相互关联的问题上的工作：让 Claude 产出高质量的前端设计，以及让它在没有人类干预的情况下构建完整的应用程序。

## 为什么简单实现会失败

我们之前展示了 harness 设计对长时间运行的 Agent 编码的有效性有重大影响。但对于更复杂的任务，Agent 仍然倾向于随着时间推移偏离轨道。我们观察到两种常见的失败模式：

1. **上下文窗口填满时模型失去连贯性** - 一些模型还表现出"上下文焦虑"，在接近上下文限制时过早结束工作
2. **自我评估问题** - 当被要求评估自己产出的工作时，Agent 倾向于自信地赞美工作——即使对人类观察者来说质量明显平庸

## 前端设计：让主观质量可评分

我从两个见解开始构建前端设计的 harness：

1. 美学不能完全简化为分数，但可以通过编码设计原则的评分标准来改进
2. 通过将前端生成与前端评分分离，我们可以创建一个反馈循环

### 四个评分标准

- **设计质量** - 设计是否感觉像一个连贯的整体？
- **原创性** - 是否有自定义决策的证据？
- **工艺** - 技术执行：排版层次、间距一致性、色彩和谐
- **功能性** - 独立于美学的可用性

我强调设计质量和原创性胜过工艺和功能性。

### 生成器-评估器循环

使用 Claude Agent SDK 构建：
1. **生成器 Agent** 创建 HTML/CSS/JS 前端
2. **评估器 Agent** 使用 Playwright MCP 与实时页面交互
3. 反馈流回生成器作为下一次迭代的输入
4. 运行 5-15 次迭代

## 扩展到全栈编码

### 三 Agent 架构

1. **规划器 (Planner)**
   - 接受 1-4 句简单提示
   - 扩展为完整的产品规格
   - 专注于产品上下文和高级技术设计

2. **生成器 (Generator)**
   - 在冲刺中工作，一次处理一个功能
   - 使用 React、Vite、FastAPI 和 SQLite/PostgreSQL 栈
   - 在每个冲刺结束时自我评估工作

3. **评估器 (Evaluator)**
   - 使用 Playwright MCP 像用户一样点击运行中的应用程序
   - 测试 UI 功能、API 端点和数据库状态
   - 对每个冲刺评分

### Sprint 合同

在每次冲刺之前，生成器和评估器协商一个 sprint 合同：在任何代码编写之前就"完成"的样子达成一致。

## 结果对比

| Harness | 持续时间 | 成本 |
|---------|----------|------|
| 单 Agent | 20 分钟 | $9 |
| 完整 Harness | 6 小时 | $200 |

Harness 贵了 20 多倍，但输出质量的差异立即可见。

### 案例研究：2D 复古游戏制作器

**单 Agent 运行**：
- 初始应用看起来符合预期
- 问题开始出现：布局浪费空间、工作流程僵化
- 实际游戏是坏的——实体出现在屏幕上但没有响应输入

**完整 Harness 运行**：
- 规划器将提示扩展为 16 个功能的规格，分布在十个冲刺中
- 应用立即显示出更多的润色和流畅性
- 画布使用整个视口，面板大小合理
- 实际上能够移动实体并玩游戏

## 简化 Harness

随着 Opus 4.6 的发布，我们开始简化 harness：

1. **移除 Sprint 结构** - 模型现在可以原生处理工作而不需要这种分解
2. **评估器改为单次通过** - 在运行结束时评估而不是每个冲刺

### DAW（数字音频工作站）案例

使用简化后的 harness，从一个简单提示开始：
> 使用 Web Audio API 在浏览器中构建一个功能齐全的 DAW

**运行统计**：
- 持续时间：约 4 小时
- 成本：$124

结果是一个功能齐全的音乐制作程序，具有：
- 工作中的编曲视图、混音器和传输控制
- 集成的 Agent 可以通过提示帮助创作音乐
- 核心的歌曲创作原语

## 下一步

随着模型继续改进，我们可以大致期望它们能够工作更长时间，处理更复杂的任务。在某些情况下，这意味着围绕模型的脚手架随着时间的推移变得越来越不重要。

但今天，精心设计的 harness 仍然是获得最佳结果的关键。**评估器不是一个固定的是或否的决定——当任务超出当前模型可以可靠地单独完成的范围时，它值得这个成本。**
