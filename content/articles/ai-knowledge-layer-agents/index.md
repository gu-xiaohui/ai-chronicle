这篇文章给你一个双层系统，让你运行的每一个 AI Agent 都更聪明。20 分钟搭建，每天都在变好，完全开源。

Karpathy 最近谈到，他正在把大部分 token 支出从代码转移到知识管理上。

这篇帖子引起共鸣，因为它点出了一个所有人都感受到的问题：**你的 Agent 不了解你。每次对话都从零开始。** 你反复解释你的业务、你的语气、你的目标、你的上下文，而输出之所以平庸，是因为输入没有记忆。

我采纳了他的模式，两周内测试了多种方案，简化出最有效的一个，并加载了我自己的数据。笔记、想法、推文、文章、书签。Agent 从中编译出 230 多个结构化 wiki 页面，用概念、实体和来源交叉引用，我可以随时查询。

本文涵盖知识层是什么、如何为内容创作、公司和个人生活搭建一个。框架已开源，搭建只需 20 分钟。

## 什么是 AI 知识层

AI 知识层是位于你和 Agent 之间的基础设施。它是 Agent 在做任何事情之前读取的内容。没有它，Agent 靠猜。有了它，Agent 知道。

它有两个部分：

1. **知识库层（KBL）**是动态的。你把原始素材丢进一个文件夹：推文、文章、书签、PDF、笔记、语音备忘录。AI Agent 读取所有内容，按类型分类，构建带交叉引用的结构化 wiki 页面，并维护一个带一行摘要的主索引供快速扫描。你问的每个问题都会被归档为新页面。wiki 随时间越来越丰富。

2. **品牌基础（BF）**是静态的。这一层只有你编辑。你的语气规则、视觉风格、定位、受众定义、你永远不用的词。Agent 在产出任何内容之前读取它，但从不改写它。它是锚点，确保即使 Agent 做了工作，一切听起来仍然像你。

为什么不用 RAG？

RAG 在查询时通过分块文档和搜索重新推导答案。知识层编译一次、交叉引用并保持更新。在约 100 篇文章时，Karpathy 发现编译方式在 Q&A 上优于 RAG。Graphify 测量到每个查询的 token 消耗比搜索原始文件**少 71.5 倍**。

演变经历了三个阶段：

1. 单次 RAG（2020-2023）
2. 带多跳检索的 Agentic RAG（2023-2024）
3. 以及现在的**上下文工程**——Agent 从多个来源构建自己的上下文（2025+）。知识层就是第三阶段的基础设施。

## 为什么大多数人不会搭建

和大多数人不会在周日备餐的原因一样。需要一小时的前期工作来节省一周十小时。大多数人宁愿抱怨 AI 输出差，也不愿花 20 分钟搭建一个修复它的系统。

**90% 使用 AI 的人现在都在做同样的事**：提示、接受、发布。没有判断，没有品味。

知识层是你离开那 90% 的方式。它编码你的品味、你的数据、你的模式，让 Agent 产出的内容听起来像你写的，而不是任何 AI 废话。

摩擦是真实的——你需要策展输入内容。如果某些东西与你正在做的事情的相关性不足 80%，跳过它。

**高信号输入带来更好的输出。填入噪音，得到的就是噪音。**

## 内容创作者和个人品牌

这就是我起步的地方。我运营一个 AI 营销内容账号和一个代理商，我需要我的 Agent 知道我写了什么、什么表现好、我的语气是什么样的。

我构建了一个叫 **LLM Wikid** 的框架并开源了（下方查看实现方式）。当我加载自己的数据后，发生了这些事：

- 我先坐下来写下脑子里所有东西：正在运行的项目笔记、半成品产品想法、对细分领域有效之处的观察、对 AI 营销走向的思考。原始思维，不关注结构。

- 然后我导入了 X 存档（6 周 87 条推文）、三篇已发表文章和所有书签。Agent 处理了一切，转化为 15 个主题源页面、14 个概念页面、11 个实体页面，它们之间有 100 多个交叉链接。

- 接着我通过 X API 拉取了 197 个书签，Agent 下载了 81 张图片并转录了 49 个视频，分析了视觉内容，将一切编织进 wiki。我的笔记、想法、推文、书签、文章，全部连接且可搜索。

## 两层架构的实际运作

**KBL（知识库层）**保存所有增长的东西：按类型分类的源（转录、文章、推文、论文），用 wikilinks 交叉引用，用 TLDR 索引。

当我需要写一个主题时，运行 `/wiki-query`，从自己的数据中获得带引用的答案。这个答案被归档为新页面，所以下次查询更丰富。

文件夹结构中，`raw/clippings/` 是 Obsidian Web Clipper 存放内容的地方。

你安装浏览器扩展，当你在浏览 X、读文章、看 GitHub repo 或刷 Reddit 时，一键剪藏。它保存到 `raw/clippings/` 并在 frontmatter 中带上源 URL。

下次运行 `/wiki-ingest`，Agent 读取 URL，检测源类型（推文、文章、PDF、视频），分类到正确的子文件夹，获取完整内容，下载图片，编译进 wiki。

**白天剪藏，夜间处理，醒来就有更丰富的知识库。**

**BF（品牌基础）**保存不变的内容：我的 AI 信号文档（列出每个被标记为 AI 的禁用词、短语和结构）、视觉内容风格指南（终端截图优先于漂亮背景，色块流程图）、我的语气档案。每个 Agent 在产出任何内容之前读取这些。

## Agent 如何接入

- **写作 Agent** 读取品牌基础获取语气，查询 wiki 做主题研究，检查内容表现数据选择正确格式。
- **研究 Agent** 监控 X、Reddit、YouTube 获取信号，将新源输入 raw 文件夹，扩展 wiki 主题。
- **内容策略师** 将细分领域中的热门表现与现有覆盖进行交叉引用，识别空白。

每个 Agent 只能和它读取的知识层一样好。

**同一个 Agent，薄知识库产出废话。同一个 Agent 读取 200 多个结构化 wiki 页面（你的语气、你的数据、你的表现历史），产出听起来像你的作品。**

我直接测试了这一点。Helena，一个 AI 营销工具，可以仅从 URL 分析品牌并产出准确的语气档案。但基于该分析生成的内容，以我的经验是"一分钟的整月营销废话"。仅品牌分析不够，你需要知识层在上面。

## 这如何映射到 AI 的 5 个层级

这连接到我之前写的 AI 营销 5 个层级：

- **Level 1**：自定义提示（没有知识层）
- **Level 2**：手动技能（薄知识层）
- **Level 3**：技能 + 品牌基础（添加 BF 层）
- **Level 4**：Agent 用技能从编译知识中读取（KBL + BF 协同工作）
- **Level 5**：自主 Agent 团队配完整复合知识层

大多数人在 Level 1 或 2。知识层让你到 4 和 5。

## 分发角度

Vibe coding 解决了构建。每个人都能在 48 小时内发布一个应用。分发仍然困难。

知识层是你复合分发智能的方式。你捕获的每个内容洞察、你分析的每个书签、你追踪的每个表现指标，都让下一篇内容更精准。

一个定时触发器每天早上对你前一天剪藏的内容运行 ingest。你的分发在你睡觉时变得更聪明。

还有一个服务角度。搭建某人的知识层是一个 $1,500-3,000 的服务，月费 $300-500。

10 个客户，第一年 $56,800。如果你是内容创作者或代理商老板，这是一个可产品化的技能。

## 公司和项目

同样的架构在公司层面也能运作。

区别在于多人的 Agent 从同一知识层读取和写入，而这一层保存的是运营知识，不仅是内容知识。

我们在 @EspressioAI 和 @LunarStrategy 运行这个。在 Espressio，知识库保存客户交付模式、Agent 架构模板和内部 SOP。

在 Lunar Strategy，它保存营销活动手册、客户研究和内容框架。新团队成员将 Agent 指向知识库，从第一天就高效运转，而不是花几周学习那些未文档化的做事方式。

Eric Osiu 在整个公司推广了类似方案。每个员工都有一个角色定制的 AI Agent 连接到共享中央大脑。他的原话：

> 当销售签下一单时，客户经理的 Agent 已经有了交接。当内容表现好时，销售 Agent 自动调整推广角度。

没有信息从裂缝中漏掉，因为知识层捕获一切。

## 如何扩展

一个人（个人 wiki）→ 小团队（5-10 人共享一个知识库）→ 组织（50+ 人配角色定制 Agent 从编译情报中读取）。每个层级的模式相同：

1. 原始源输入
2. Agent 编译结构化页面
3. 交叉引用自动构建
4. 人类通过探索门验证（每个页面开始标记为未审查，只有人类标记为已确认）
5. Git 同步一切，知识有版本控制和可回滚

Cody Schneider 为本地服务业务构建了 10 能力 GTM Agent 集群。他系统的基础是一个保存所有业务数据的**数据仓库**。

他叫它数据仓库。Eric Osiu 叫它共享大脑。Karpathy 叫它 LLM wiki。

名字不重要。Agent 需要编译的、结构化的知识来做有用的工作。

这个的极端版本是 Greg Isenberg 所说的"环境业务"——主要靠 Agent 运行的公司，所有者每隔几天看一眼。

Medvi 是证明：18 亿美元收入运行率，2 名员工，零 VC。AI 处理代码、创意、语音、客服。知识层是前提条件。没有它，Agent 需要持续的人类指导。有了它，它们从编译的组织情报中运转。

## 个人生活

最被低估的用途。追踪内容表现的同一个系统可以追踪你的思考。

日记条目、读书笔记、播客高亮、健康指标、目标回顾、凌晨两点的随机想法。

全部进入 `raw/`。Agent 编译为结构化页面。

当你问"我的精力水平有什么规律"或"上季度我学到了什么关于生产力的东西"，你从自己的数据获得带引用的答案。这个答案被归档，所以下一个问题更聪明。

**复合循环：你问的每个问题让系统更丰富。你添加的每个源创建新连接。你的好奇心反哺你获得的答案质量。**

质量控制在这里比任何地方都重要，因为这是盲目信任 AI 最危险的地方：

- **偏差检查**强制每个页面有反面论据和数据缺口。如果你导入 10 篇都说同样话的文章，wiki 会注意到但也会标记画面中缺失了什么。
- **验证门**意味着每个 AI 生成的页面以 `explored: false` 开始。只有你能标记为已审查。你始终知道什么经过了人类验证，什么没有。
- **置信度**为每个页面标记为高、中、低或不确定。Agent 必须诚实说明其知识有多可靠。

我"三个护城河"文章中的 80/20 法则直接适用：让 AI 做 80% 的组织、编译和交叉引用。你的品味用在最后 20%——策展、验证、只有你能看到的连接。**知识层是你编码品味的方式，让那 80% 随时间变得更好。**

## 20 分钟搭建指南

这不是理论。框架已开源，搭建很快。

**Step 1：克隆仓库（2 分钟）**

将文件夹作为 Obsidian vault 打开。

**Step 2：运行 Agent（3 分钟）**

在文件夹中打开 Claude Code（或任何能读 markdown 和运行 bash 的 Agent）。它读取 `CLAUDE.md`——控制一切的 schema。

Agent 读取这个文件，理解完整 schema，自动脚手架 wiki 结构。

**Step 3：填入数据（10 分钟）**

- 从设置请求你的 X 存档
- 获取 Obsidian Web Clipper 扩展，开始剪藏推文、文章、线程
- 坐下来写 10 分钟脑子里所有东西。倒入书签。任何感觉 80%+ 相关的放进 `raw/`

**Step 4：运行 ingest（5 分钟）**

Agent 分类你的剪藏，从 URL 获取完整内容，下载并分析图片，按类型分类每个源，构建带交叉引用的 wiki 页面，为每个页面添加反面论据和数据缺口，更新主索引。

**Step 5：打开图谱**

打开 Obsidian 的图谱视图。你的想法、你关注的人、你用的工具、你关心的概念，现在连接在一个视觉网络中。用 `/wiki-query` 开始查询。

初始 ingest 完成后，试试这些：每个答案都会被归档，wiki 会增长。

## 现在该做什么？

设置一个定时触发器（Claude Dispatch 或 cron），每天早上运行 `/wiki-ingest`。

你白天剪藏，夜间处理，醒来就有更丰富的知识库。

每 1-2 周运行 `/wiki-lint` 捕捉页面间的矛盾、过时内容、没人链接的孤立页面、不同名称下的重复概念。

Agent 修复能修的，标记需要你判断的。

当达到 300+ 页面时，安装 qmd（Tobi Lutke 开发）做本地混合搜索——BM25 + 向量检索 + LLM 重排。它有 MCP server，你的 Agent 可以搜索 wiki 作为原生工具。

开始将知识层连接到其他 Agent。从中读取的写作 Agent。向其中输入的研究 Agent。查询它的内容策略师。**知识层是让所有这些有用的共享大脑。**

## 窗口期

Karpathy 的帖子获得了 99,000+ 书签。Graphify 在 48 小时内发布并获得 27,000+。多个实现同一周内走红。需求是显而易见的。

大多数人会收藏这个，想"挺酷"，然后永远不搭建。

**今天花 20 分钟的人，下个月就会有一个复合知识库，没有任何搜索引擎和通用 AI 提示能复制。它会知道你的语气、你的数据、你的模式和你的品味。**

你等的一周，就是一周的复合你拿不回来。
