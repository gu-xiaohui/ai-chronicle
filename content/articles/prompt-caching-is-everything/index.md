工程界常说 "Cache Rules Everything Around Me"，同样的规则也适用于 Agent。

像 Claude Code 这样的长期运行 Agent 产品之所以可行，是因为 **Prompt Caching（提示词缓存）**，它允许我们重用之前轮次的计算，显著降低延迟和成本。

什么是 Prompt Caching，它是如何工作的，如何从技术上实现它？

## 为缓存布局你的 Prompt

![Prompt 布局](/images/articles/prompt-caching-is-everything/image-2.jpg)

Prompt Caching 通过**前缀匹配**工作 —— API 缓存从请求开始到每个 `cache_control` 断点的所有内容。这意味着你放置内容的顺序非常重要，你希望尽可能多的请求共享前缀。

**最好的方法是：静态内容在前，动态内容在后。**

对于 Claude Code，这看起来像：

1. **静态系统提示** & 工具（全局缓存）
2. **CLAUDE.MD**（在项目内缓存）
3. **会话上下文**（在会话内缓存）
4. **对话消息**

这样我们可以最大化有多少会话共享缓存命中。

但这可能非常脆弱！我们之前打破这个顺序的原因包括：在静态系统提示中放置详细的时间戳、非确定性地打乱工具顺序定义、更新工具参数（例如 AgentTool 可以调用什么 Agent）等。

## 使用消息进行更新

有时候你放在 prompt 中的信息会过时，例如如果你有时间或者用户更改了文件。你可能会想更新 prompt，但这会导致缓存未命中，并且可能对用户来说相当昂贵。

考虑是否可以通过**下一轮的消息**传入这些信息。在 Claude Code 中，我们在下一个用户消息或工具结果中添加 `<system-reminder>` 标签，包含模型的更新信息（例如现在是星期三），这有助于保留缓存。

## 不要在会话中途更换模型

Prompt 缓存是**特定于模型**的，这会让 prompt caching 的计算变得非常不直观。

如果你在与 Opus 的对话中已经使用了 100k token，想问一个相当容易回答的问题，实际上切换到 Haiku 比 让 Opus 回答**更昂贵**，因为我们需要为 Haiku 重建 prompt 缓存。

如果你需要切换模型，最好的方法是使用 **子 Agent**，Opus 会准备一个"交接"消息给另一个模型，说明需要完成的任务。我们在 Claude Code 中经常这样做，Explore agent 使用 Haiku。

## 永远不要在会话中途添加或删除工具

在对话中途更改工具集是人们破坏 prompt caching 最常见的方式之一。这看起来很直观 —— 你应该只给模型它现在需要的工具。但因为工具是缓存前缀的一部分，添加或删除工具会使整个对话的缓存失效。

### Plan Mode — 围绕缓存设计

Plan Mode 是围绕缓存约束设计功能的一个很好的例子。直观的方法是：当用户进入计划模式时，换掉工具集只包含只读工具。但这会破坏缓存。

**相反，我们保持所有工具**在请求中始终存在，并使用 `EnterPlanMode` 和 `ExitPlanMode` 作为工具本身。当用户切换计划模式时，Agent 会收到一条系统消息，解释它处于计划模式以及指令是什么 —— 探索代码库，不要编辑文件，计划完成后调用 ExitPlanMode。工具定义永远不会改变。

这有一个额外的好处：因为 `EnterPlanMode` 是模型自己可以调用的工具，它可以在检测到困难问题时**自主进入计划模式**，而不会破坏缓存。

### Tool Search — 延迟而不是移除

同样的原则适用于我们的工具搜索功能。Claude Code 可以加载几十个 MCP 工具，在每次请求中包含所有这些工具会很昂贵。但在对话中途移除它们会破坏缓存。

**我们的解决方案：`defer_loading`**。我们发送轻量级的存根 —— 只是工具名称，带有 `defer_loading: true` —— 模型可以通过 `ToolSearch` 工具在需要时"发现"它们。完整的工具模式只有在模型选择它们时才加载。这使缓存的前缀保持稳定：相同的存根始终以相同的顺序存在。

## 分叉上下文 — Compaction（压缩）

![Compaction](/images/articles/prompt-caching-is-everything/image-3.jpg)

Compaction 是当你用尽上下文窗口时发生的事情。我们总结到目前为止的对话，并用该摘要继续一个新的会话。

令人惊讶的是，compaction 与 prompt caching 有许多边缘情况，可能不太直观。

特别是，当我们压缩时，我们需要将整个对话发送给模型以生成摘要。如果这是一个单独的 API 调用，使用不同的系统提示且没有工具（这是简单的实现），主对话的缓存前缀根本不匹配。你需要为所有这些输入 token 支付全价，大大增加了用户的成本。

### 解决方案 — 缓存安全的分叉

当我们运行 compaction 时，我们使用与父对话**完全相同**的系统提示、用户上下文、系统上下文和工具定义。我们在前面加上父对话的消息，然后在末尾附加 compaction 提示作为新的用户消息。

从 API 的角度来看，这个请求看起来与父请求几乎相同 —— 相同的前缀、相同的工具、相同的历史 —— 所以缓存的前缀被重用。唯一的新 token 是 compaction 提示本身。

但这确实意味着我们需要保存一个"compaction buffer"，以便我们在上下文窗口中有足够的空间包含压缩消息和摘要输出 token。

## 经验总结

1. **Prompt caching 是前缀匹配。** 前缀中任何位置的任何更改都会使其后的所有内容失效。围绕这个约束设计你的整个系统。把顺序弄对，大部分缓存就会自动工作。

2. **使用消息而不是系统提示更改。** 你可能会想编辑系统提示来做像进入计划模式、更改日期等事情，但实际上最好在对话期间将这些插入到消息中。

3. **不要在对话中途更改工具或模型。** 使用工具来建模状态转换（如计划模式），而不是更改工具集。延迟工具加载而不是移除工具。

4. **像监控正常运行时间一样监控缓存命中率。** 我们会对缓存中断发出警报，并将它们视为事件。几个百分点的缓存未命中率可能会极大地影响成本和延迟。

5. **分叉操作需要共享父级的前缀。** 如果你需要运行侧计算（compaction、摘要、技能执行），使用相同的缓存安全参数，以便在父级前缀上获得缓存命中。

Claude Code 从第一天起就围绕 prompt caching 构建，如果你正在构建 Agent，你也应该这样做。
