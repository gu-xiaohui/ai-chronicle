目前 GitHub 上排名第一的仓库是一个超级代理 harness——字节跳动的 DeerFlow。它将记忆存储在 JSON 中。以下是它的工作原理。

大多数框架通过回放聊天历史来模拟记忆，将所有内容塞进上下文窗口，希望模型能挑出重要的东西。

**DeerFlow 做法不同。**它不存储对话。它提取关于用户的事实，按置信度评分，并将适合 2,000 token 预算的内容注入到每个提示中——异步地，不会使上下文膨胀。

在本文中，我将基于逆向工程我在 Docker 中运行的自托管 DeerFlow 实例并检查其 JSON 记忆来详细解析其工作原理。

## 什么是 DeerFlow

DeerFlow（深度探索和高效研究流程）是一个开源的超级代理 harness，它协调子代理、记忆和沙箱来做几乎任何事情，由可扩展的技能驱动。它目前在 GitHub 上以 49.2K 星排名第一。

![记忆面板](/images/articles/deerflow-memory/memory-panel.jpg)

DeerFlow UI 看起来像 ChatGPT 或 Claude 界面，左侧有聊天界面和代理选项。我在 MacBook 上的 Docker 容器中本地运行，进行了约 3 小时的对话测试。

**测试内容：**
- 与 DeerFlow 讨论技术话题：记忆基准（LOCOMO、LongMemEval）、理解 DeerFlow 架构，以及我的爱好（滑翔伞推荐）
- 通过提问并要求代理忽略某些记忆来测试记忆检索

没有得到足够关注的部分是**记忆层**。

## 记忆界面

记忆面板将所有内容组织为：

- **用户上下文**（工作 + 个人）
- **当前关注点**（首要事项）
- **历史层**（近几个月、早期上下文、长期背景）
- **事实表**

你可以看到系统学到的关于你的一切、学习时间以及置信度。每个事实上的来源字段是实际的线程 UUID，因此系统确切知道每条知识来自哪次对话。

## 记忆存储在哪里

DeerFlow 中的记忆以文件形式存在于磁盘上：`backend/.deer-flow/memory.json`。它是本地的，跨每个会话持久化。一个结构化的 JSON 文件。

![memory.json](/images/articles/deerflow-memory/memory-json.jpg)

在一个会话中，DeerFlow 构建了一个结构化的配置文件。每个事实都有与记忆面板 UI 中看到的类似内容。**置信度低于 0.7 的事实不会被包含。**存储上限为 100 条事实，溢出时首先淘汰最低置信度的。

## 实际工作原理

DeerFlow 的主代理运行在 LangGraph 上，这是一个基于图的编排框架，其中每个代理轮次都是有状态执行图中的一个节点。

**记忆不是插件或 sidecar。它直接嵌入到中间件链中作为第 8 位**，意味着它在每个代理轮次上自动运行。

关键组件是 `MemoryMiddleware`。

![中间件链](/images/articles/deerflow-memory/middleware.png)

MemoryMiddleware 位于 TitleMiddleware 之后、视觉和循环检测层之前。这是有意为之的——记忆更新应该在标题生成之后（这样 LLM 知道对话内容）但在任何循环或澄清检查中断会话之前。

**MemoryMiddleware 不同步更新记忆。它将对话排队进行异步处理。**

![写入流程](/images/articles/deerflow-memory/write-flow.png)

流程如下：

1. 用户发送消息。代理响应。
2. MemoryMiddleware 过滤交互——只考虑用户输入和最终 AI 响应。
3. 对话被添加到异步队列，带有 30 秒的去抖计时器。如果 30 秒内任何线程有新消息到达，计时器重置。
4. LLM 提取器对对话运行并生成差异：`newFacts`（要添加）、`factsToRemove`（要删除）和每个摘要部分的 `shouldUpdate` 标志。
5. 记忆 JSON 更新。

在附加新事实之前，系统检查精确的内容重复（通过去除空格进行规范化）。注意：相同语义事实的不同表述仍然会被添加——去重是基于文本的，不是语义的。

你可以将 `model_name` 指向更便宜的模型进行提取步骤——记忆 LLM 不需要是你最好的模型，它只需要擅长结构化提取。

![配置](/images/articles/deerflow-memory/queue.png)

在第 3 步中我发现有趣的是，如果同一个 `thread_id` 已经有待处理的更新在队列中，新条目会替换它而不是追加。你永远不会处理过时的对话中间快照——只有每个线程的最终状态会被处理。

记忆摄入不会影响用户获取代理响应的延迟体验。如果用户发送消息，他们会立即得到响应。这是一个良好代理 harness 的示例。

![去抖时间线](/images/articles/deerflow-memory/injection.jpg)

## 记忆实际出现在哪里

当你开始新对话时，每个适合 2,000 token 预算的事实（按置信度排序），加上所有用户、历史和首要事项摘要，都会被注入到系统提示中的 `<memory>` 标签内。

没有硬编码的"前 N"限制——这是一个 token 预算，不是计数。Tiktoken 在事实逐条添加时精确计算 token，直到预算用完。

代理在每个会话开始时看到类似这样的内容。代理永远不需要问"你在做什么？"——它已经基于我的首要事项历史知道了。

## 记忆检索测试

我尝试了更多测试来了解它是否能根据对话检索我的信息。检索按预期工作。

![检索检查](/images/articles/deerflow-memory/retrieval.png)

我尝试通过聊天删除记忆。它只在事实表中添加了更多偏好——尽管 LLM 提取器步骤显示它有 `factsToRemove` 差异记录。

![删除失败](/images/articles/deerflow-memory/delete.png)

这绝对是有限制的，可以通过修改代码库中的 json 来解决。这显示了此记忆系统的局限性。

另一个限制是当我在某次会话中谈论记忆基准时，尽管我的用户配置显示英语偏好，它显示的问题推荐却是中文。

## 结论

### DeerFlow 记忆亮点

- **本地且属于你。** 记忆存储没有供应商锁定。它是一个 JSON 文件。你可以读取、编辑、重置它。
- **默认异步。** 记忆更新永远不会减慢你的响应。30 秒去抖很聪明——它吸收对话噪音而不会在每条消息上消耗 LLM 调用。
- **置信度评分存储。** 一次提到某事（"我可能会尝试 Rust"）不会永久改变你的记忆配置。置信度阈值在弱信号积累之前过滤掉它们。
- **原子写入。** 先写后重命名是正确的模式。进程崩溃时的记忆损坏是大多数玩具实现忽略的真实故障模式。
- **去重。** 附加前的重复检查意味着系统不会以 50 种不同表述层叠同一个偏好 50 次。

### 它做不到的事

- **没有语义搜索。** 注入的是适合 token 预算的最高置信度事实，而不是与当前查询最相关的事实。如果你有 100 条存储的事实并问一个高度具体的问题，你得到的是按置信度排名的顶级事实，而不是与该问题相关的事实。
- **没有语义去重。** 重复检查是基于文本的——它去除空格并精确比较字符串。我可以在实际记忆文件中看到：关于比较 DeerFlow 记忆与文件系统 + markdown 方法的事实——相同语义意图，不同表述，存储为两个独立条目。一次会话后 9 条事实不是问题。一个月后 80 条事实时就开始重要了。
- **有限的智能层 + 潜在的压缩问题。** 记忆上限为 100 条事实，这意味着随着上下文增长它可能会吃力。虽然置信度分数有助于优先考虑保留或省略哪些记忆。
- **没有向量嵌入。** 这是一个有意的权衡——没有嵌入管道，没有相似性搜索。它保持系统简单且依赖少，但也带来了限制。

## 为什么这很重要

大多数代理框架将记忆视为**检索问题**：嵌入所有内容，存储在向量数据库中，在查询时检索最相似的块。这可行，但有代价。你在每个请求上添加了嵌入模型、向量存储和检索步骤。这意味着更多基础设施、更多延迟和更多复杂性。

**DeerFlow 采取了不同方法：不存储对话，存储理解。** 它运行异步 LLM 提取通道，将原始对话提炼为结构化事实，然后将这些事实直接注入到未来的提示中。昂贵的步骤在响应已经交付之后发生，在去抖的 30 秒计时器上，使用你选择的任何模型。

总的来说，作为一个超级代理 Harness 深度研究和代理工具，DeerFlow 团队在这次发布中实现记忆系统做得很好。结果是一个**了解你的代理**，不是因为它记住你说的所有内容，而是因为它构建了关于你的偏好、目标和上下文的持久模型。而且那个模型作为简单的 JSON 文件存在于你的机器上。

DeerFlow 将记忆直接构建到其中间件中，所以它开箱即用，在每个对话上，默认启用。这是值得关注的部分。
