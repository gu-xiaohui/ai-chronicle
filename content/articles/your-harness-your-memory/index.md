Agent Harness 正成为构建 Agent 的主流方式，而且它们不会消失。这些 Harness 与 Agent 记忆深度绑定。如果你使用闭源 Harness——尤其是基于私有 API 的——你实际上是将 Agent 记忆的控制权交给了第三方。记忆对创建优质且具粘性的 Agent 体验至关重要。这创造了极强的锁定效应。**记忆——以及由此带来的 Harness——应该是开放的，这样你才能真正拥有自己的记忆。**

## Agent Harness 是构建 Agent 的方式，而且不会消失

过去三年里，构建 Agent 系统的"最佳方式"发生了翻天覆地的变化。

ChatGPT 刚出来时，你只能做简单的 RAG 链（LangChain）。然后模型变好了一点，能创建更复杂的流程（LangGraph）。然后模型变得好得多，催生了一种新的脚手架——**Agent Harness**。

Agent Harness 的例子包括 Claude Code、Deep Agents、Pi（驱动 OpenClaw）、OpenCode、Codex、Letta Code 等等。

**Agent Harness 不会消失。**

有时会有一种观点认为模型会吸收越来越多的脚手架。这不是真的。发生的事情是（并将继续发生）：2023 年需要的大量脚手架不再需要了。但这被其他类型的脚手架取代了。Agent，按定义，就是一个与工具和其他数据源交互的 LLM。模型周围将始终有一个系统来促进这种交互。

需要证据？当 Claude Code 的源代码泄露时，有 **51.2 万行代码**。这些代码就是 Harness。即使是世界上最好的模型制造者也在大力投资 Harness。

当网络搜索等功能被内置到 OpenAI 和 Anthropic 的 API 中时——它们不是"模型的一部分"。它们是一个轻量级 Harness 的一部分，位于它们的 API 后面，通过工具调用协调模型与这些网络搜索 API。

## Harness 与记忆深度绑定

Letta CTO Sarah Wooders 写了一篇[很棒的博客](https://x.com/sarahwooders/status/2040121230473457921)，解释"记忆不是插件（它就是 Harness）"，我完全同意。

有时会有一种观点认为记忆是一个独立的服务，与任何特定 Harness 分开。在目前这个阶段，这不是事实。

Harness 的大部分职责是与上下文交互。正如 Sarah 所说：

> 请求将记忆插入 Agent Harness，就像请求将驾驶插入汽车一样。管理上下文，因此管理记忆，是 Agent Harness 的核心能力和责任。

**记忆只是上下文的一种形式。**短期记忆（对话中的消息、大型工具调用结果）由 Harness 处理。长期记忆（跨会话记忆）需要由 Harness 更新和读取。

Sarah 列出了 Harness 与记忆绑定的许多其他方式：

- AGENTS.md 或 CLAUDE.md 文件如何加载到上下文中？
- 技能元数据如何展示给 Agent？（在系统提示中？在系统消息中？）
- Agent 能否修改自己的系统指令？
- 什么在压缩（compaction）中存活，什么会丢失？
- 交互是否被存储并可供查询？
- 记忆元数据如何呈现给 Agent？
- 当前工作目录如何表示？暴露了多少文件系统信息？

**记忆作为一个概念目前还处于婴儿期。**长期记忆通常不是 MVP 的一部分。首先你需要让 Agent 一般性地工作，然后才能考虑个性化。这意味着我们（作为一个行业）仍在摸索记忆。目前还没有广为人知的通用记忆抽象。

## 如果你不拥有你的 Harness，你就不拥有你的记忆

Harness 与记忆深度绑定。如果你使用闭源 Harness，尤其是基于 API 的，你不拥有你的记忆。

这以几种方式表现出来：

**轻微的坏处：** 如果你使用有状态 API（如 OpenAI 的 Responses API 或 Anthropic 的服务端压缩），你在将状态存储在他们的服务器上。如果你想切换模型并恢复之前的线程——那已经不可能了。

**严重的坏处：** 如果你使用闭源 Harness（如 Claude Agent SDK，底层使用不开源的 Claude Code），这个 Harness 以你未知的方式与记忆交互。也许它在客户端创建了一些制品——但那些制品的形态是什么，Harness 应该如何使用它们？这是未知的，因此无法从一个 Harness 转移到另一个。

**最坏的：** 当整个 Harness，包括长期记忆都在 API 后面时。在这种情况下，你对记忆——包括长期记忆——**零所有权、零可见性**。你不知道 Harness（这意味着你不知道如何使用记忆）。更糟的是——你甚至不拥有记忆本身！

当人们说"模型会吸收越来越多的 Harness"时——他们真正的意思就是这个。他们指的是这些与记忆相关的部分会进入模型提供商提供的 API 后面。

**这极其令人担忧**——它意味着记忆将被锁定在单一平台、单一模型上。

模型提供商有极强的动机这样做。而且他们已经开始这样做了。Anthropic 推出了 Claude Managed Agents。这实际上将所有东西都放在 API 后面，锁定在他们的平台上。

即使整个 Harness 不在 API 后面，模型提供商也有动机将越来越多的东西移到 API 后面——而且已经在这样做了。例如：尽管 Codex 是开源的，它生成加密的压缩摘要（在 OpenAI 生态系统之外不可用）。

**他们为什么这样做？因为记忆很重要，它创造了仅靠模型无法获得的锁定。**

## 记忆很重要，它创造了锁定

虽然记忆还处于早期，但所有人都清楚它的重要性。它使 Agent 能够随着用户交互而变得更好，并允许你建立数据飞轮。它使你的 Agent 能够为每个用户个性化，构建一个适应他们愿望和使用模式的 Agent 体验。

**没有记忆，你的 Agent 很容易被任何能接触到相同工具的人复制。**

有了记忆，你就建立了一个专有数据集——一个用户交互和偏好的数据集。这个专有数据集使你能够提供差异化的、越来越智能的体验。

到目前为止，切换模型提供商相对容易。它们有相似甚至相同的 API。当然，你需要稍微改一下提示词，但这不难。

但这都是因为它们是无状态的。

一旦有任何关联的状态，切换就难多了。因为这个记忆很重要。如果你切换，你就会失去对它的访问。

让我讲个故事。我内部有一个邮件助手。它构建在我们 Fleet 平台上的模板之上。这个平台内置了记忆功能，所以过去几个月我与之交互时，它积累了记忆。几周前，我的 Agent 被意外删除了。我很生气！我尝试用相同的模板创建一个 Agent——但体验差太多了。我不得不重新教它所有偏好、语气、一切。

我的邮件 Agent 被删除的一个好处——它让我意识到记忆可以多么强大、多么有粘性。

## 开放记忆，开放 Harness

记忆需要被开放，由开发 Agent 体验的人拥有。它允许你建立一个你真正控制的专有数据集。

记忆（以及由此带来的 Harness）应该与模型提供商分离。你应该有选项性来尝试最适合你用例的任何模型。模型提供商有动机通过记忆创造锁定。

这就是为什么我们在构建 Deep Agents。Deep Agents：

- **开源**
- **模型无关**
- 使用 [agents.md](http://agents.md/) 和 [skills](https://agentskills.io/home/) 等开放标准
- 有 Mongo、Postgres、Redis 等插件的记忆存储
- 可通过 LangSmith 部署或自行托管，可部署在任何云上
- 可以自带数据库作为记忆存储
- 可以部署在任何标准 Web 托管框架后

**为了拥有你的记忆，你需要使用开放 Harness。**
