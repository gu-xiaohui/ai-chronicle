我们 99% 的生产代码由 AI 编写。上周二，我们上午 10 点发布了一个新功能，中午 A/B 测试，下午 3 点因为数据不好就砍掉了，下午 5 点发布了一个更好的版本。三个月前，这样的周期需要六周。

我们不是靠在 IDE 里加个 Copilot 到达这个状态的。**我们拆解了整个工程流程，围绕 AI 重建了它。** 我们改变了规划、构建、测试、部署和团队组织的方式。我们改变了公司里每个人的角色。

CREAO 是一个 Agent 平台，25 名员工，10 名工程师。我们从 2025 年 11 月开始构建 Agent，两个月前我从头重构了整个产品架构和工程流程。

OpenAI 在 2026 年 2 月发布了一个概念，精准地描述了我们一直在做的事。他们称之为 **Harness 工程**：工程团队的首要工作不再是写代码，而是让 Agent 能够做有用的工作。当某个环节失败时，解决方案从来不是"再试一次"，而是：缺少了什么能力？如何让 Agent 能够理解并遵循这个能力？

我们独立得出了同样的结论，当时还没有人给它命名。

## AI-First ≠ 用了 AI

大多数公司只是把 AI 贴在现有流程上。工程师打开 Cursor，PM 用 ChatGPT 写需求文档，QA 试用 AI 测试生成。工作流没变，效率提升 10-20%，没有任何结构性变化。

那叫 **AI 辅助**。

**AI-First** 意味着你围绕"AI 是主要构建者"这一假设，重新设计流程、架构和组织。你不再问"AI 如何帮助工程师？"，而是问"我们如何重组一切，让 AI 来构建，工程师来提供方向和判断？"

这个差异是乘法级的。

我看到很多团队声称 AI-First，但还在跑同样的 Sprint 周期、同样的 Jira 看板、同样的周会站会、同样的 QA 签字确认。他们把 AI 加进了循环，但没有重新设计循环本身。

一个常见变体是人们说的"氛围编程"（vibe coding）：打开 Cursor，提示直到能用，提交，重复。这能产出原型。但生产系统需要稳定、可靠、安全。你需要一个能保证这些属性的系统——即使代码是 AI 写的。**你构建的是系统，提示词是可抛弃的。**

## 为什么我们必须改变

去年，我观察团队的工作方式，发现了三个会杀死我们的瓶颈。

### 产品管理的瓶颈

我们的 PM 花几周时间研究、设计、定义功能。产品管理几十年来都是这样工作的。但 Agent 两小时就能实现一个功能。当构建时间从几个月压缩到几小时，几周长的规划周期就成了约束。

花几个月思考，然后两小时构建——这说不通。

PM 需要进化为以产品为导向的架构师，以迭代速度工作；或者退出构建循环。设计需要通过快速的"原型-发布-测试-迭代"循环来实现，而不是委员会审查的需求文档。

### QA 的瓶颈

同样的动态。Agent 交付功能后，QA 团队花几天测试边界情况。构建时间：两小时。测试时间：三天。

我们用 AI 构建的测试平台替代了手动 QA，用 AI 构建的测试来验证 AI 写的代码。验证速度必须与实现速度匹配，否则你只是在旧瓶颈下游十英尺处建了一个新瓶颈。

### 人数的瓶颈

我们的竞争对手有 100 倍以上的人做类似的工作。我们只有 25 人。我们无法通过招聘来追平，必须通过重新设计来追平。

三个系统都需要 AI 贯穿其中：产品如何设计、产品如何实现、产品如何测试。任何一个保持手动，都会约束整个管线。

## 大胆的决定：统一架构

我必须先修复代码库。

旧架构分散在多个独立系统中。一个变更可能需要同时改三四个仓库。从人类工程师的角度看，这还可以管理。从 AI Agent 的角度看，是不透明的。Agent 看不到全貌，无法推理跨服务的影响，无法在本地运行集成测试。

**我必须把所有代码统一到一个 monorepo。原因之一：让 AI 能看到一切。**

这是一个 Harness 工程原则的实践：你把系统的越多部分暴露为 Agent 可检查、可验证、可修改的形式，你获得的杠杆就越大。碎片化的代码库对 Agent 是隐形的，统一的代码库是可读的。

我花一周设计了新系统：规划阶段、实现阶段、测试阶段、集成测试阶段。然后用 Agent 花另一周重新架构了整个代码库。

CREAO 是一个 Agent 平台。我们用自己的 Agent 重建了运行 Agent 的平台。如果产品能构建自己，说明它工作正常。

## 技术栈

### 基础设施：AWS

我们在 AWS 上运行，使用自动伸缩容器服务和断路器回滚。如果部署后指标恶化，系统自动回滚。

CloudWatch 是中枢神经系统。所有服务的结构化日志，超过 25 个告警，自动化工作流每天查询自定义指标。基础设施的每一部分都暴露结构化的、可查询的信号。如果 AI 读不了日志，它就诊断不了问题。

### CI/CD：GitHub Actions

每次代码变更都通过六阶段管线：

验证 CI → 构建并部署 Dev → 测试 Dev → 部署 Prod → 测试 Prod → 发布

每个 PR 的 CI 门禁强制执行类型检查、lint、单元和集成测试、Docker 构建、通过 Playwright 的端到端测试和环境一致性检查。没有可选阶段，没有人工跳过。管线是确定性的，Agent 可以预测结果并推理失败原因。

### AI 代码审查：Claude

每个 PR 触发三个并行的 AI 审查，使用 Claude Opus 4.6：

- **Pass 1**：代码质量——逻辑错误、性能问题、可维护性
- **Pass 2**：安全——漏洞扫描、认证边界检查、注入风险
- **Pass 3**：依赖扫描——供应链风险、版本冲突、许可证问题

这些是审查门禁，不是建议。它们与人工审查并行运行，在大量 PR 中捕获人类遗漏的问题。当你每天部署八次，没有人类审查员能在每个 PR 上保持注意力。

工程师还在任何 GitHub issue 或 PR 中 @claude，获取实现计划、调试会话或代码分析。Agent 看到整个 monorepo，上下文在对话间保持。

### 自愈反馈循环（核心）

每天 UTC 9:00 AM，一个自动化健康工作流运行。Claude Sonnet 4.6 查询 CloudWatch，分析所有服务的错误模式，生成执行层健康摘要，通过 Microsoft Teams 送达团队。没有人要求它这样做。

一小时后，分诊引擎运行。它从 CloudWatch 和 Sentry 聚类生产错误，在九个严重性维度上评分，并在 Linear 中自动创建调查工单。每个工单包含样本日志、受影响用户、受影响端点和建议的调查路径。

系统自动去重。如果已有 open issue 覆盖同一错误模式，它更新该 issue。如果已关闭的 issue 复发，它检测回归并重新打开。

当工程师推送修复时，同样的管线处理它。三个 Claude 审查 pass 评估 PR。CI 验证。六阶段部署管线逐步推进 dev 和 prod 并在每阶段测试。部署后，分诊引擎重新检查 CloudWatch。如果原始错误已解决，Linear 工单自动关闭。

每个工具处理一个阶段。没有工具试图做所有事。这个日常循环创建了一个**自愈循环**：错误被检测、分诊、修复和验证，最少的人工干预。

我告诉 Business Insider 的记者："AI 会创建 PR，人类只需要审查是否存在风险。"

### Feature Flags 和支撑栈

Statsig 处理功能开关。每个功能都在开关后面发布。发布模式：先对团队启用，然后渐进式百分比发布，然后全量发布或终止。终止开关可以瞬间关闭功能，无需部署。如果功能指标恶化，我们在几小时内拉掉它。**坏功能在发布当天就会死。** A/B 测试通过同一系统运行。

Graphite 管理 PR 分支：合并队列 rebase 到 main，重新运行 CI，仅绿时合并。Stacked PR 允许在高吞吐量下增量审查。

Sentry 报告所有服务的结构化异常，由分诊引擎与 CloudWatch 合并以获得跨工具上下文。Linear 是人类层面：自动创建的工单包含严重性评分、样本日志和建议调查。

## 一个功能如何从想法到生产

### 新功能路径

1. 架构师将任务定义为结构化提示，包含代码库上下文、目标和约束
2. Agent 分解任务、规划实现、编写代码并生成自己的测试
3. PR 创建。三个 Claude 审查 pass 评估它。人类审查员检查战略风险，而非逐行正确性
4. CI 验证：类型检查、lint、单元测试、集成测试、端到端测试
5. Graphite 合并队列 rebase、重新运行 CI、绿时合并
6. 六阶段部署管线逐步推进 dev 和 prod 并在每阶段测试
7. 功能开关对团队开启。渐进式百分比发布。指标监控
8. 如有退化可终止。严重问题自动断路器回滚

### Bug 修复路径

1. CloudWatch 和 Sentry 检测到错误
2. Claude 分诊引擎评分严重性，创建 Linear issue，包含完整调查上下文
3. 工程师调查。AI 已经完成了诊断。工程师验证并推送修复
4. 同样的审查、CI、部署和监控管线
5. 分诊引擎重新验证。如果已解决，工单自动关闭

两条路径使用同一管线。一个系统。一个标准。

## 结果

14 天内，我们平均每天 3-8 次生产部署。在旧模式下，整个两周周期连一次生产发布都做不到。

坏功能在发布当天就被拉掉。新功能在构想当天就上线。A/B 测试实时验证影响。

人们以为我们在用质量换速度。用户参与度上升了。支付转化率上升了。我们产出了比以前更好的结果，因为反馈循环更紧密。每天发布比每月发布学到的更多。

## 新的工程组织

工程师将只有两种。

### 架构师

一到两个人。他们设计教 AI 如何工作的标准操作流程。他们构建测试基础设施、集成系统、分诊系统。他们决定架构和系统边界。他们为 Agent 定义"好"的标准。

这个角色需要深度批判性思维。你批评 AI，不跟随它。当 Agent 提出计划时，架构师找到漏洞。什么失败模式被遗漏了？什么安全边界被跨越了？什么技术债务在积累？

我有物理学博士学位。博士教给我最有用的东西是如何质疑假设、压力测试论证、发现缺失的部分。**批评 AI 的能力将比写代码的能力更有价值。**

这也是最难招聘的角色。

### 操作员

其他所有人。工作很重要，结构不同。

AI 给人类分配任务。分诊系统发现 bug，创建工单，呈现诊断结果，分配给合适的人。这个人调查、验证、批准修复。AI 创建 PR。人类审查是否存在风险。

任务是 bug 调查、UI 细化、CSS 改进、PR 审查、验证。它们需要技能和注意力，但不需要旧模式要求的架构推理能力。

### 谁适应最快

我观察到了一个意想不到的模式：**初级工程师比高级工程师适应得更快。**

传统实践较少的初级工程师感到被赋能了。他们获得了放大影响力的工具。他们没有十年的习惯需要抛弃。

传统实践较强的高级工程师适应最难。他们两个月的工作可以被 AI 一小时完成。在多年建立稀有技能集之后，这很难接受。

我不是在做评判，我是在描述我观察到的事实。在这个转型中，**适应力比积累的技能更重要。**

## 人的一面

### 管理坍塌

两个月前，我 60% 的时间在管理人。对齐优先级、开会、给反馈、辅导工程师。

现在：不到 10%。

传统 CTO 模式说要赋能团队做架构工作、培训他们、委派。但如果系统只需要一两个架构师，我需要先自己做。我从管理变成了构建。大多数日子我从早上 9 点写到凌晨 3 点。我设计 SOP 和系统架构。我维护 Harness。

压力更大。但我享受构建，而不是对齐。

### 更少争论，更好的关系

我与联合创始人和工程师的关系比以前更好。

转型前，我与团队的大部分互动是对齐会议。讨论取舍、争论优先级、不同意技术决策。这些对话在传统模式中是必要的，但也很消耗。

现在我仍然和团队交谈，但我们聊的是其他事情。非工作话题、随意对话、团建旅行。我们相处得更好了，因为我们不再争论系统可以轻松完成的工作。

### 不确定性是真实的

我不会假装每个人都很开心。

当我不再每天和人说话时，一些团队成员感到不确定。CTO 不和我说话意味着什么？我在新世界中的价值是什么？这些是合理的担忧。

有些人花更多时间争论 AI 是否能做他们的工作，而不是去做工作。转型期制造焦虑。我没有一个干净的答案。

但我有一个原则：我们不会因为工程师引入了生产 bug 就解雇他。我们改进审查流程、加强测试、增加护栏。**同样的原则适用于 AI。** 如果 AI 犯了错，我们构建更好的验证、更清晰的约束、更强的可观测性。

## 超越工程

我看到其他公司采用 AI-First 工程但其他一切保持手动。

如果工程几小时就能发布功能但营销需要一周来宣布，营销就是瓶颈。如果产品团队还在跑月度规划周期，规划就是瓶颈。

在 CREAO，我们将 AI 原生操作推进到每个职能：

- 产品发布说明：AI 从 changelog 和功能描述生成
- 功能介绍视频：AI 生成的动效图形
- 社交媒体日常帖子：AI 编排并自动发布
- 健康报告和分析摘要：AI 从 CloudWatch 和生产数据库生成

工程、产品、营销和增长在一个 AI 原生工作流中运行。如果一个职能以 Agent 速度运行而另一个以人类速度运行，人类速度的职能会约束一切。

## 这意味着什么

### 对工程师

你的价值正在从代码产出转移到决策质量。快速写代码的能力每个月都在贬值。评估、批评和指导 AI 的能力在升值。

**产品品味很重要。** 你能看一个生成的 UI 就知道它在用户投诉之前就是错的吗？你能看一个架构提案就发现 Agent 遗漏的失败模式吗？

我告诉我们的 19 岁实习生：训练批判性思维。学习评估论证、发现漏洞、质疑假设。学习好设计长什么样。这些技能会复利。

### 对 CTO 和创始人

如果 PM 流程比构建时间长，从那里开始。

在扩展 Agent 之前先构建测试 Harness。没有快速验证的快速 AI 是快速移动的技术债务。

从一个架构师开始。一个人构建系统并证明它能工作。系统运行后再让其他人进入操作员角色。

将 AI 原生推进到每个职能。预期会有阻力。

### 对行业

OpenAI、Anthropic 和多个独立团队收敛到了同样的原则：结构化上下文、专业化 Agent、持久记忆和执行循环。Harness 工程正在成为标准。

模型能力是驱动这一切的时钟。我将 CREAO 的整个转变归因于最近两个月。Opus 4.5 做不到 Opus 4.6 能做的事。下一代模型会进一步加速。

**我相信一人公司会变得普遍。** 如果一个架构师加上 Agent 能做 100 人的工作，很多公司不需要第二个员工。

## 我们还很早

我和大多数创始人和工程师交谈时，他们仍然按传统方式运作。有些在想转型，极少数已经做了。

一个记者朋友告诉我，她在这个话题上大约和五个人聊过。她说我们比任何人都走得远："我不认为有任何人像你们这样完全重建了整个工作流程。"

工具已经存在，任何团队都可以这样做。我们的技术栈中没有什么是专有的。

竞争优势在于围绕这些工具重新设计一切的决策，以及吸收成本的意愿。成本是真实的：员工的不确定性、CTO 每天 18 小时工作、高级工程师质疑自己的价值、旧系统已消失但新系统尚未证明的两周过渡期。

我们吸收了那个成本。两个月后，数据说话了。

**我们构建一个 Agent 平台。我们用 Agent 构建了它。**
