构建 Agent 框架最困难的部分之一是构建其**工具空间（Action Space）**。

Claude 通过工具调用（Tool Calling）来行动，考虑到所有这些选项，你如何设计 Agent 的工具？你只需要一个工具吗？如果你有 50 个工具，每种使用场景一个，会怎样？

## 工具设计的框架

为了让自己进入模型的心态，我喜欢想象被给到一个困难的数学问题。为了解决它，你需要什么工具？这取决于你自己的技能！

- **纸** 是最低要求，但你会受限于手动计算
- **计算器** 会更好，但你需要知道如何操作更高级的选项
- **电脑** 是最快和最强大的选择，但你需要知道如何使用它

这是设计 Agent 的一个有用框架。**你想给它与其能力相匹配的工具。**

但你怎么知道这些能力是什么？你要**关注、阅读它的输出、实验**。你要**学会像 Agent 一样思考**。

## 教训 1：AskUserQuestion 工具

我们尝试了三种方式来改进 Claude 的提问能力：

### 尝试 #1 - 修改 ExitPlanTool

在 ExitPlanTool 中添加参数来包含问题数组。但这让 Claude 感到困惑。

### 尝试 #2 - 修改输出格式

让 Claude 输出特定格式的问题。但 Claude 不总是遵循格式。

### 尝试 #3 - AskUserQuestion 工具 ✅

创建一个 Claude 可以在任何时候调用的工具。**最重要的是，Claude 似乎喜欢调用这个工具。**

即使设计得最好的工具，如果 Claude 不理解如何调用它，也不会起作用。

## 教训 2：Tasks 取代 Todos

当我们首次推出 Claude Code 时，我们给了 Claude TodoWrite 工具。

但随着模型的改进，它们不仅不需要被提醒 Todo 列表，而且可能会觉得它有限制性。

所以我们用 **Task 工具**替换了 TodoWrite。Tasks 更多是关于帮助 Agent 相互通信。

**随着模型能力的增加，你的模型曾经需要的工具现在可能会限制它们。**

## 教训 3：搜索界面

当 Claude Code 首次推出时，我们使用 RAG 向量数据库为 Claude 查找上下文。**Claude 是被给予这个上下文，而不是自己找到上下文。**

通过给 Claude 一个 Grep 工具，我们可以让它搜索文件并自己构建上下文。

**这是我们随着 Claude 变得更聪明而看到的一个模式：如果给它正确的工具，它变得越来越擅长构建自己的上下文。**

## 教训 4：渐进式披露

我们引入了**渐进式披露（progressive disclosure）**的想法，它允许 Agent 通过探索逐步发现相关的上下文。

在一年中，Claude 从不太能够构建自己的上下文，发展到能够跨多层文件进行嵌套搜索。

**渐进式披露现在是我们用来添加新功能而不添加工具的常见技术。**

## 结语

> 经常实验，阅读你的输出，尝试新事物。像 Agent 一样思考。

设计 Agent 的工具既是艺术也是科学。它取决于你使用的模型、Agent 的目标以及它运行的环境。
