## 1: AGENT 如何工作

![Agent 工作原理](/images/articles/build-ai-agent-full-course/how-agents-work.jpg)

这是所有 Agent 共享的核心循环：

**用户输入 → LLM 思考 → LLM 决策（回应或调用工具）→ 如果工具：执行并反馈结果 → 重复**

- **LLM** 是推理的"大脑"
- **工具** 是执行动作的"手"（计算器、网络搜索、文件读写）
- **记忆** 是记录发生事情的"笔记本"

无论你使用 LangGraph、CrewAI、Anthropic SDK 还是 OpenAI Agents SDK，这些框架用抽象包装了这个循环，但**不改变其本质**。

**增强型 LLM**

普通 LLM 接受文本并发出文本。增强型 LLM 添加三个能力：

- **工具**：模型可以调用的函数（计算器、数据库、API、文件操作等）
- **检索**：从外部源（搜索引擎、文档、向量数据库）提取相关信息的能力
- **记忆**：通过消息历史或其他持久存储在交互间保留信息的能力

**工作流 vs 真正的 Agent**

工作流是确定性的——你的代码控制执行，相同输入始终产生相同路径。适合明确定义的任务，成本更低。

Agent 是动态的——LLM 决定下一步，可能反复调用工具。适合开放式任务，但成本更高。

**建议：从简单工作流开始，然后看是否需要升级为自主 Agent。**

## 2: 五种核心工作流模式

![五种模式](/images/articles/build-ai-agent-full-course/five-workflows.jpg)

大多数问题实际上不需要完全自主。这五种由 Anthropic 记录的模式涵盖常见用例：

**模式 1：提示链**
将任务分解为顺序步骤。每个 LLM 调用处理前一个的输出。在步骤之间添加程序化"门"来验证质量。

**模式 2：路由**
分类输入，然后路由到专门的处理器。每个处理器有自己的优化提示。

**模式 3：并行化**
同时运行多个 LLM 调用。分段将任务分成独立子任务并行处理。投票对同一任务运行多次并聚合结果。

**模式 4：编排器-工作者**
中央 LLM 动态分解任务并委派给工作者 LLM。与并行化不同，子任务不是预定义的——编排器在运行时决定。

**模式 5：评估器-优化器**
一个 LLM 生成输出，另一个评估并提供反馈。如果评估失败，反馈循环回去。重复直到满足质量标准。

## 3: 构建你的 AGENT

![构建 Agent](/images/articles/build-ai-agent-full-course/building-agent.jpg)

如何将"我想让 Agent 做 XYZ"变成现实？最简单的思考方式：

1. 写下任务
2. 决定需要什么工具
3. 告诉模型如何行为
4. 在 5 个真实示例上测试
5. 只在失败时添加更多复杂性

**最简单的心智模型：回答四个问题**

1. **结果是什么？** Agent 应该实际产出什么？
2. **它需要什么信息？** 网络搜索、文件、数据库、还是只需用户消息？
3. **它应该被允许采取什么行动？** 只回答？搜索？编辑文件？发送邮件？
4. **它必须遵循什么规则？** 语气、格式、约束、安全规则、不确定时怎么办。

**如果你能清楚地回答这四个问题，通常一天内就能构建出第一版。**

**初学者友好的 Agent 设计公式：**

Agent = 角色 + 目标 + 工具 + 规则 + 输出格式

**五种初学者 Agent 类型：**

1. **研究 Agent** — 收集信息并总结
2. **内容 Agent** — 写作、重写、总结或转换内容
3. **工作流 Agent** — 遵循可重复的业务流程
4. **个人知识 Agent** — 使用你的文档回答问题
5. **操作 Agent** — 在环境中执行操作

**Anthropic 适合**：需要读写文件、使用 shell 命令、网络搜索、MCP 工具、编码任务的 Agent。

**OpenAI 适合**：干净的 API、自定义函数工具、托管工具、Agent 间移交、护栏和追踪。

## 4: 使用工具

![工具](/images/articles/build-ai-agent-full-course/tools.jpg)

**更多工具 ≠ 更聪明的 Agent。更好的工具 = 更聪明的 Agent。更少的工具 = 更可靠的 Agent。**

工具就是"AI 无法自己完成的事情"：计算数字、搜索网络、读取文件、发送邮件、查询数据库。

**关键规则：**

- 如果需要外部数据或操作 → 使用工具
- 如果不需要 → 不要添加
- 一个工具 = 一个明确的任务
- 告诉 Agent **何时**使用工具

**坏工具设计**：`manage_files(action, file, destination, overwrite, format, permissions)`

**好工具设计**：
```python
read_file(path)
write_file(path, content)
delete_file(path)
```

## 5: 给 AGENT 记忆

![记忆](/images/articles/build-ai-agent-full-course/memory.jpg)

**只有两种类型的记忆：**

1. **短期记忆（对话）**：到目前为止说了什么——默认就有
2. **长期记忆（外部知识）**：Agent 以后可以查找的东西——笔记、PDF、文档、数据库

**什么时候需要记忆？**

- Agent 需要跨消息记住内容？→ 短期记忆
- 需要使用外部文档？→ 长期记忆
- 否则 → 可能不需要

**三个选项：**

- 选项 A：无记忆（从这里开始，70% 的用例够用）
- 选项 B：对话记忆（大多数 SDK 已处理）
- 选项 C：基于文件的记忆（简单 RAG）

**核心规则：如果你的 Agent 不需要记忆就能工作 → 不要添加。**

## 6: 让 AGENT 在现实中工作

![现实工作](/images/articles/build-ai-agent-full-course/making-work.jpg)

Agent 要么好用，要么不好用，大多数不好用是因为：提示词差、没有测试、不切实际的期望。

**测试准则：**

- 不要测试："请分类此技术问题"
- 要测试："我的账户坏了而且一直在收费我该怎么办"

像真实用户一样测试，修复一件事，用 AI 调试 Agent。

**不要过早添加复杂性。**

## 7: 多 AGENT 系统

![多 Agent](/images/articles/build-ai-agent-full-course/multi-agent.jpg)

**更多 Agent ≠ 更强大。始终从一个 Agent 开始。**

只在以下情况添加更多：
- 任务明显分离
- 一个 Agent 在挣扎
- 角色非常不同

**需要多 Agent 的三种情况：**
1. 不同技能（研究 Agent + 写作 Agent）
2. 清晰的流水线（输入 → 分析 → 写作 → 输出）
3. 不同权限（一个读数据，一个执行操作）

**最安全的模式：监督者模型**
用户 → 主 Agent → （需要时调用其他 Agent）

## 8: 总结

本指南最重要的洞察是：**Agent 概念上简单但操作上有要求。** 核心循环（LLM 思考、调用工具、重复）适合 50 行 Python。真正的工作在于工具设计、错误处理、评估，以及知道何时更简单的模式会优于自主 Agent。

**三个可操作的要点：**

1. **先构建从零开始的 Agent。** 理解原始循环让每个框架变得透明而不是神奇。你会更快调试问题并更明智地选择工具。

2. **从最简单的模式开始。** 提示链处理大多数多步骤任务。路由模式处理大多数分类-然后-行动工作流。只在需要 LLM 动态决定执行路径时升级到自主 Agent。

3. **尽早投资工具设计和评估。** 设计良好的工具——清晰的名称、精确的描述和结构化的错误消息——将比切换模型或框架更能提高 Agent 性能。20 个好的测试用例比任何数量的手动测试能发现更多 bug。

**你现在可以构建 Agent 了。**
