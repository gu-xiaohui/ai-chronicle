你已经用 OpenClaw 自动化了开发工作流。但每次查询的成本比应有的更高。

你的代理扫描 50 个文件来回答一个问题。读取了 10,000 个 token。实际需要的只有 300 个。

这不是检索问题。**这是结构问题。**

OpenClaw 将记忆存储在每日 markdown 文件中。每个会话一个文件。三个月后，你有 90 个文件。当你问"我们是如何实现支付流程的？"时，代理会扫描所有文件来寻找相关上下文。**它读取所有内容来找到任何东西。**

你可以手动将重要知识整理到 MEMORY.md 中使其持久化，但那个文件会随着时间变得臃肿。你在手动维护第二个知识库。

核心问题不是 OpenClaw 缺乏记忆。**而是记忆没有结构。**

```
~/.openclaw/workspace-dev/
├── MEMORY.md         # 精心整理的长期记忆（持久事实、决策、偏好）
└── memory/
    ├── 2026-02-19.md  # 今天的日志
    ├── 2026-02-18.md  # 昨天的日志（会话开始时自动加载）
    └── [older dates].md # 历史每日日志
```

## OpenClaw 记忆目前的运作方式

OpenClaw 使用**语义向量搜索**来检索记忆。它将你的 markdown 文件分成约 400 token 的片段，带有 80 token 的重叠，生成嵌入，并存储在 SQLite 中。当你提问时，它搜索语义相似的片段并返回最接近的匹配。

这对基本回忆有效。但**相似性不等于相关性**。

向量搜索找到听起来相似的文本。它不理解概念之间的关系。它无法区分三个月前的关键架构决策和昨天的一条随意评论。

结果：跨项目污染、压缩期间上下文消失，以及在不相关文件上浪费 token 的检索。

## 结构化记忆是什么样的

不是扁平的每日日志，记忆应该**分层组织**。把它想象成归档系统。领域在顶层。主题在领域内。子主题在主题内。

当新信息进来时，代理应该推理它在层次结构中的位置并将其放在那里。**而不是附加到今天的日志文件中。**

这从根本上改变了检索。代理不是搜索 90 个文件来找相似文本，而是直接导航到正确的主题。

ByteRover 为 OpenClaw 实现了这种架构。它现在已合并到主仓库。

## 三层架构

![架构](/images/articles/byterover-openclaw-memory/architecture.png)

ByteRover 在 OpenClaw 现有系统之上添加了第三层记忆：

**上下文树（ByteRover）** — 持久化、LLM 策展的结构化知识
存储如下信息：
> "支付流程使用 Stripe webhooks 进行异步确认（在 src/payments/webhooks.ts 中实现，处理 3 种事件类型）"

**工作区记忆（MEMORY.md）** — 持久化、手动策展的规则
存储偏好如：
> "使用 TypeScript 严格模式。所有 API 响应必须包含错误代码。"

**每日记忆（memory/YYYY-MM-DD.md）** — 仅追加的会话日志
存储活动如：
> "2026-03-15: 重构了认证中间件。仍需添加速率限制。"

**当代理开始会话时：**
- MEMORY.md 告诉它如何行为
- 昨天的日志告诉它上次停在哪里
- 上下文树告诉它对项目了解什么

**上下文树是使这不同的关键。** 它不是被动存储。LLM 代理主动策展信息，确定什么重要，并在存储之前进行组织。

## 可检查的记忆

每次策展产生结构化反馈。当 ByteRover 向上下文树添加知识时，你可以确切看到发生了什么：

- 添加了什么
- 更新或合并了什么与现有知识
- 跳过了什么以及原因
- 什么失败了（缺失数据、冲突、不可用来源）

这创造了**可审计性**。你可以追踪信息来自哪里、何时学习的，以及每个决策背后的推理。

## 不扫描一切的检索

![检索](/images/articles/byterover-openclaw-memory/retrieval.jpg)

ByteRover 使用**分层检索管道**而不是一次性向量搜索：

**缓存查找 → 全文搜索 → LLM 驱动搜索**

- **频繁查询**立即命中缓存。没有 LLM 调用，没有文件扫描。
- **不频繁查询**使用结构化树的全文本搜索。快速、有针对性，没有昂贵的推理。
- **需要推理的复杂查询**作为最后手段触发 LLM 驱动搜索。

我在一个 1,000+ 文件的项目上测试了这个，每天 10 个编码问题。标准向量搜索在冗余读取上燃烧了数十万 token。**ByteRover 将其减少了 83%。**

## 基准测试

ByteRover 在 LoCoMo 基准上达到了 **92.19% 的检索准确率**，经过 8 个月以上的架构迭代。这是目前市场上最高的。

差异来自前端的 LLM 策展而不是被动的嵌入存储。

记忆是**本地优先**的。它作为可读的 markdown 文件存在于你的机器上。你可以像代码一样编辑、重构和版本控制它。当你需要在其他地方使用时，推送到 ByteRover 的云端。多个 OpenClaw 代理可以共享相同的记忆。你的桌面代理和笔记本电脑代理自动保持同步。

## 如何设置

一条命令：

```bash
curl -fsSL https://www.byterover.dev/install.sh | sh
```

ByteRover 与 OpenClaw 现有记忆并行工作。你控制什么被策展。

核心命令：

- `brv query` — 从上下文树检索
- `brv curate` — 添加或更新知识
- `brv push / brv pull` — 与云端同步

每次策展产生结构化反馈。你可以确切看到什么被添加、更新、合并或跳过了，以及原因。

## 这改变了什么

**记忆变得可移植。** 从 OpenClaw 到 Claude Code 到 Cursor 到下一个代理框架——你的上下文随你而行。

**上下文树是人类可读的。** 打开任何文件，查看代理知道什么、何时学到的以及如何组织的。

如果代理需要跨会话记忆而不在文件扫描上燃烧 token，**这就是解决方案。**
